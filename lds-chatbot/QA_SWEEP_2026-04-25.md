## Executive Summary

**This demo is NOT demo-ready for clients today.** The production deployment is fundamentally broken in two independent, P0-severity ways:

1. **Every client-side `fetch("/api/...")` call 404s** because the calls do not include the configured `NEXT_PUBLIC_BASE_PATH=/agents`. The Next.js `basePath` config does NOT auto-rewrite plain `fetch()` URLs — it only affects `next/link` and `useRouter`. As a result, MCP/Specialist/Dataset/Agent CRUD UI flows ALL fail silently (no error toast on submit; the dialog just stays open). Lists show "Failed to load X" toast on every page that fetches client-side. The Edit Agent page renders "Agent not found" even when the agent exists.
2. **The chat workflow on the platform produces empty assistant content.** Even with a valid model slug (`gpt-4.1-mini`) and the OpenAI-compatible API path called directly (which bypasses bug #1), the platform completes the workflow with `finish_reason: "stop"` and zero content chunks. `usage.prompt_tokens=0, completion_tokens=0`. The chat is fundamentally non-functional from any path.

There is no demo flow that can be run successfully end-to-end in the current state. Sign-in works; viewing empty SSR-rendered list pages works; literally anything else is broken.

The DB was found completely empty on arrival (no pre-seeded "Code du travail", "Légifrance", or "KALI" specialists/MCPs that the prompt referenced). The fix mentioned in the prompt for the Specialists MCP-name-display bug could not be verified because no specialists exist — and creating one through the UI is impossible due to bug #1.

**Quick fix path**: The basePath bug is mechanical — wrap every client-side `fetch("/api/...")` call (16+ call sites in `app/(app)/**/*.tsx`) with the basePath. The empty-content bug requires investigation against the platform Backend — possibly the workflow graph being built in `lib/platform/workflows.ts` references model slugs/IDs incorrectly.

## Severity-Ranked Bug List

### P0 - Critical (broken core functionality)

#### P0-1: All client-side `fetch("/api/...")` calls 404 — basePath not prepended
**Repro**:
1. Sign in, navigate to https://demo.alien.club/agents/mcps
2. Observe toast "Failed to load MCP servers"
3. Click "Add MCP", fill form, click Add
4. Dialog stays open, no error feedback, MCP is NOT created

**Expected**: GET `/agents/api/mcps` returns 200 with the MCP list; POST creates the MCP and dialog closes with success toast.
**Actual**: Browser hits `https://demo.alien.club/api/mcps` (no `/agents` basePath) → 404. Same for POST.

**Evidence**:
- Network log: `GET https://demo.alien.club/api/mcps => 404`, `POST https://demo.alien.club/api/mcps => 404`
- Verified the API itself works fine when called with the correct path: `fetch('/agents/api/mcps')` returns 200/201.
- ConfigMap confirms `NEXT_PUBLIC_BASE_PATH=/agents` is set.

**Affected pages (all of these are broken)**:
- `app/(app)/mcps/page.tsx` lines 221, 232, 246 — `/api/mcps`, `/api/mcps/${id}`
- `app/(app)/agents/new/page.tsx` lines 40, 61 — `/api/models`, `/api/agents`
- `app/(app)/agents/[id]/page.tsx` lines 149, 150, 151, 300 — `/api/models`, `/api/specialists`, `/api/mcps`, `/api/specialists`
- `app/(app)/agents/[id]/chat/page.tsx` lines 21, 31 — `/api/agents/${agentId}`, `api: "/api/chat"`
- `app/(app)/agents/[id]/chat/[conversationId]/existing-chat-client.tsx` line 32 — `api: "/api/chat"`
- `app/(app)/datasets/page.tsx` line 50 — `/api/datasets`
- `app/(app)/datasets/new/page.tsx` line 76 — `/api/datasets`
- `app/(app)/datasets/[id]/page.tsx` line 213 — `/api/agents`
- `app/(app)/specialists/new/page.tsx` lines 49, 50, 77 — `/api/models`, `/api/mcps`, `/api/specialists`
- `app/(app)/specialists/[id]/page.tsx` lines 70, 71 — `/api/models`, `/api/mcps`

**Suspected fix**: Either centralize fetches behind a helper that prepends `process.env.NEXT_PUBLIC_BASE_PATH`, or use Next's `useRouter().basePath` everywhere, or set `NEXT_PUBLIC_APP_URL` and use absolute URLs. Note that `next.config.ts` currently sets `basePath` based on `NEXT_PUBLIC_BASE_PATH` but Next's basePath does NOT apply to `fetch()` — it only applies to navigations.

#### P0-2: Chat / agent workflow returns empty assistant content
**Repro**:
1. Create an agent with a valid model slug (e.g. `gpt-4.1-mini`) via the API:
   ```
   POST /agents/api/agents { name, systemPrompt, model: "gpt-4.1-mini", ... }
   ```
2. Hit the OpenAI-compat endpoint (which works because it uses an absolute path inside Next):
   ```
   POST /agents/api/v1/agent/{id}/v1/chat/completions
     { messages: [{role:"user", content:"say HELLO"}], model:"agent", stream:true }
   ```
3. Observe the SSE stream: 2 chunks total — initial `{role:"assistant", content:""}` and `finish_reason:"stop"` — never any actual content.

**Expected**: The agent streams text deltas with the model's response.
**Actual**: Stream completes with no content. Non-streaming response: `content: ""`, `prompt_tokens: 0`, `completion_tokens: 0`.

**Evidence**:
- Server logs show successful workflow run on platform: `POST https://api.alien.club/workflows/18/run → 200`, `GET https://api.alien.club/jobs/35/stream → 200`. Stream connects but no chunks come through with any content.
- Confirmed with both `mistral-small-latest` (the default — note: this slug does NOT exist on the platform; correct slug would be `mistral-small-2603`) and `gpt-4.1-mini` (which IS in the LLM list).
- Both stream and non-stream variants reproduce.
- Same 0-tokens result regardless of model.

**Suspected location**: This may be a platform-side workflow execution issue rather than a chatbot code issue, since the chatbot just consumes whatever chunks the platform's `/jobs/:id/stream` SSE emits. However, it's possible the workflow graph built in `lib/platform/workflows.ts` is missing required `deepAgent` parameters (e.g. provider config, API keys, or model-ID-vs-slug mismatch). Worth verifying that `deepAgent-4` node's `model` param accepts a slug and that the workflow graph is wired with correct edges to produce a `result.stream.agent.chunks` array.

#### P0-3: No error feedback on UI form submit failures
**Repro**:
1. Open Add MCP dialog, fill in valid data, click Add
2. Observe: dialog stays open, no toast, no console error

**Expected**: Either the MCP appears or an error toast like "Failed to create MCP".
**Actual**: Silent failure. The fetch wrapper in the create handlers (e.g. `app/(app)/mcps/page.tsx`) does have a `try/catch`, but the 404 returns HTML (Next.js 404 page). The `await response.json()` may succeed on the HTML payload as `{}`, then `response.ok` is false but the error path doesn't show a toast clearly. Result: user has no idea why nothing happened.

**Suspected file**: `app/(app)/mcps/page.tsx` create handler around line 246. Same pattern in `agents/new/page.tsx` line 61, `specialists/new/page.tsx` line 77.

### P1 - High (feature broken, workaround exists)

#### P1-1: Sign-out redirects to non-existent URL (404)
**Repro**:
1. Click the sign-out button (icon next to user avatar in sidebar)
2. Browser is taken to https://demo.alien.club/sign-in → 404 ("Looks like there's a problem with this site, Error code: 404 Not Found")

**Expected**: Redirect to `/agents/sign-in` (with basePath).
**Actual**: 404 page. Manually navigating to `/agents/sign-in` works fine.

**Evidence**: Page URL after sign-out click was `https://demo.alien.club/sign-in`. The sign-in page itself at `/agents/sign-in` works.

**Suspected file**: Likely the sign-out handler in `lib/auth.ts` or wherever the sign-out callback URL is configured. The `redirectTo` for sign-out is hardcoded without basePath.

#### P1-2: `/api/models` does not filter by `modelType=llm` — returns all 95 models
**Repro**:
```
GET /agents/api/models
```
**Expected**: ~53 LLM models only.
**Actual**: 95 models including 20 TTS, 6 embedding, 9 image-generation, 7 video-generation. The query string `?select=public&modelType=llm` is ignored by the platform Backend.

**Evidence**: Direct call returned `{tts:20, llm:53, embedding:6, "image-generation":9, "video-generation":7}`.

**Why it matters for the chatbot**: The agent creation form uses model slug as a free-text fallback when models list is empty (it 404s due to P0-1 anyway). If the basePath bug is fixed, this filter bug means the agent creator dropdown will be polluted with TTS and embedding models that won't work as LLMs.

**Suspected location**: Likely a Backend issue (`api.alien.club /ai-models` endpoint) — outside this repo. But the chatbot should also defensively client-filter to `modelType === "llm"` after fetching.

#### P1-3: Default model slug `mistral-small-latest` does not exist on the platform
**Repro**: Inspect the LLM list — Mistral models present are `mistral-small-2603`, `mistral-large-2512`, etc. There is no `mistral-small-latest` slug.
**Expected**: Default to a model that actually exists.
**Actual**: New agents and subagents default to a slug with no matching model on the platform; if the workflow then silently no-ops on missing model (which appears to be P0-2's symptom), every default-config agent will be DOA.

**Suspected files**: `app/(app)/agents/new/page.tsx` line 37, `app/api/agents/[id]/route.ts` line 69, `lib/db/schema.ts` (default value), `app/api/agents/[id]/route.ts` line 76.

#### P1-4: No messages persisted from OpenAI-compat path
**Repro**: Hit `/agents/api/v1/agent/{id}/v1/chat/completions` 4 times. Check `messages` table → 0 rows. Check `conversations` table → 4 rows (one per request).
**Expected**: User and assistant messages persisted; same conversation reused for the same `conversation_id` round-trip.
**Actual**: The OpenAI-compat route only persists `session_id` (line 96 `persistSessionId`), never inserts into `messages`. Each call also creates a new conversation row (auto-titled `openai-api`) instead of associating with an existing thread when no `conversation_id` is passed.

**Suspected file**: `app/api/v1/agent/[id]/v1/chat/completions/route.ts` — needs to insert user message before run and assistant message after stream completes, mirroring the logic in `app/api/chat/route.ts`.

### P2 - Medium (UI glitches, dead UX)

#### P2-1: PUT `/api/agents/{id}` with `steps` as a string (not array) crashes server
**Repro**:
```
PUT /agents/api/agents/{id} { ...existingAgent, model: "x" }
```
where `existingAgent.steps` is the string `"[]"` (as returned by the GET endpoint).
**Expected**: 400 Bad Request with a clear validation error.
**Actual**: 500 Internal Server Error with no body. Server logs show `TypeError: t.map is not a function`.

**Evidence**: Server log:
```
⨯ TypeError: t.map is not a function
    at <unknown> (.next/server/chunks/[root-of-the-server]__04f3f8m._.js:1:8142)
    ...
```

**Suspected file**: `app/api/agents/[id]/route.ts` line 68 (`body.steps ?? (existing.steps ? JSON.parse(existing.steps) : [])`) — when `body.steps` IS provided but is a string, the fallback parse is skipped and the string flows into `buildAgentWorkflow` which calls `.map()` on it. Needs validation. Also: GET should return `steps` as array (parsed), not as JSON string, to make round-trip safe.

#### P2-2: `/api/agents/{id}` does not support PATCH
**Repro**: `PATCH /agents/api/agents/{id} { model: "x" }` → 405 Method Not Allowed.
**Expected**: Either PATCH is supported for partial updates, or there's a specific validation message.
**Actual**: 405 with empty body. Only PUT works (and PUT requires the entire body).

**Suspected file**: `app/api/agents/[id]/route.ts` — only exports GET, PUT, DELETE.

#### P2-3: Misleading "Agent not found" message on Edit Agent page
**Repro**: Navigate to `/agents/agents/{valid-existing-agent-id}`.
**Expected**: Edit form populated with agent data.
**Actual**: Shows "Agent not found." with a back link, plus a toast "Failed to load agent". The agent does exist; the page just couldn't fetch it (basePath bug).

**Suspected file**: `app/(app)/agents/[id]/page.tsx` — error path collapses "fetch failed" with "404 not found".

#### P2-4: Dialog missing `aria-describedby` accessibility attribute
**Repro**: Open Add MCP dialog. Console warning: `Warning: Missing 'Description' or 'aria-describedby={undefined}' for {DialogContent}.`
**Expected**: All shadcn/ui Dialogs should have `<DialogDescription>` for screen readers.
**Actual**: Missing — accessibility regression. Likely affects all dialogs in the app.

**Suspected file**: Various Dialog usages, e.g. `components/mcp-dialog.tsx` (or wherever the MCP dialog is defined).

## Final Take

This is a fresh deploy where the most recent code that hit production introduced a regression covering EVERY client-side fetch. There is no end-to-end happy path that works in the UI today. The fixes are mostly mechanical (P0-1 is a global find-and-replace), but the empty-content chat bug (P0-2) needs platform-side investigation — and that's the demo's reason for existing.

**Do not show this to a client today.** Estimated time to demo-ready: 0.5–1 day for the basePath sweep and sign-out fix, plus however long it takes to track down why the platform workflow returns no chunks (could be a few hours, could be a day or more depending on whether it's the chatbot's workflow graph or the Backend's deepAgent execution).
