# i18n Rule

## Rule

The codebase uses `next-intl` with `messages/en.json` and `messages/fr.json`. Every user-visible string in JSX is a translation key. No exceptions.

---

## Key Requirements

**Both locale files must stay in sync.** A key present in `en.json` but absent from `fr.json` is a broken build waiting to happen. Every time you add a key to one file, you add it to both files in the same commit.

**French is the default locale** (`as-needed` strategy in `next.config.ts`). When adding a feature, translate it properly — placeholder strings like `"TODO"` or copied English text in `fr.json` are not acceptable.

---

## Namespace Structure

Namespaces mirror the feature domain, not the file path. The top-level key is the feature; nested keys describe the page or component within that feature.

```
{
  "agents": {
    "page": { "title": "…", "createButton": "…" },
    "card": { "chatButton": "…", "editButton": "…", "deleteButton": "…" },
    "form": { "namePlaceholder": "…", "promptLabel": "…" }
  },
  "datasets": {
    "page": { "title": "…", "uploadButton": "…" },
    "status": { "pending": "…", "processing": "…", "ready": "…", "error": "…" }
  },
  "common": {
    "save": "…",
    "cancel": "…",
    "delete": "…",
    "loading": "…",
    "error": "…",
    "tryAgain": "…"
  }
}
```

### Established top-level namespaces

| Namespace | Covers |
|---|---|
| `agents` | Agent list, detail, creation, cards |
| `datasets` | Dataset list, upload, status badges |
| `specialists` | Specialist list and cards |
| `mcps` | MCP server CRUD |
| `conversations` | Conversation list and chat UI |
| `auth` | Sign-in page |
| `nav` | Sidebar and navigation labels |
| `common` | Shared across features: save, cancel, delete, loading, error, tryAgain |

Do not introduce a new top-level namespace unless you are adding a wholly new feature domain. Shared cross-feature labels go in `common`.

---

## Key Naming

Keys are **camelCase**. They name the concept, not the string value — `chatButton`, not `chatWithAgent` or `click_here_to_chat`.

```json
// ✅ Correct
"card": {
  "chatButton": "Chat",
  "deleteConfirm": "Are you sure you want to delete {name}?"
}

// ❌ Wrong — snake_case
"card": {
  "chat_button": "Chat"
}

// ❌ Wrong — describes the value, not the concept
"card": {
  "clickHereToChatWithThisAgent": "Chat"
}
```

---

## Scoping `useTranslations`

Always scope to the **smallest relevant namespace**. A component that only renders agent cards should not hold the entire `agents` namespace — it should scope to `agents.card`.

```tsx
// ✅ Correct — scoped to the component's namespace
export function CardAgentSummary({ agent }: { agent: Agent }) {
  const t = useTranslations("agents.card")
  const tCommon = useTranslations("common")

  return (
    <Card>
      <CardTitle>{agent.name}</CardTitle>
      <Button>{t("chatButton")}</Button>
      <Button variant="destructive">{tCommon("delete")}</Button>
    </Card>
  )
}

// ❌ Wrong — root-level useTranslations pulls the entire message tree
const t = useTranslations()

// ❌ Wrong — page-level namespace used inside a card component
const t = useTranslations("agents")
// then: t("card.chatButton") — defeats the scoping purpose
```

## Pattern: Adding a New Feature

When adding a new page or component with user-visible text:

1. Identify the feature namespace (`agents`, `datasets`, etc.)
2. Identify the component sub-namespace (`page`, `card`, `form`, `dialog`)
3. Add all keys to `messages/en.json` under `feature.component.*`
4. Add all keys to `messages/fr.json` in the same location with proper French translations
5. Use `useTranslations("feature.component")` in the component — never the root `useTranslations("feature")`

```json
// messages/en.json
{
  "specialists": {
    "page": {
      "title": "Specialists",
      "emptyMessage": "No specialists yet.",
      "createButton": "New Specialist"
    }
  }
}

// messages/fr.json  ← must be updated in the same commit
{
  "specialists": {
    "page": {
      "title": "Agents spécialisés",
      "emptyMessage": "Aucun agent spécialisé pour l'instant.",
      "createButton": "Nouvel agent"
    }
  }
}
```

## Anti-Patterns (FORBIDDEN)

```tsx
// ❌ Hardcoded English string in JSX
<Button>Create Agent</Button>

// ❌ Hardcoded string in any user-facing attribute
<Input placeholder="Enter a name" />

// ❌ Root-level useTranslations
const t = useTranslations()

// ❌ Flat top-level key with no namespace
// messages/en.json: { "createAgentButton": "Create" }

// ❌ Key exists in en.json but not fr.json
// Adding to en.json and shipping without updating fr.json
```

---

## Scope

This rule covers all files under:

- `app/` (pages, layouts, error boundaries)
- `components/` (all components including shadcn wrappers)

Dynamic string values from the database (agent names, dataset names, user content) are not translation keys and are rendered as-is. The rule governs static UI chrome — labels, buttons, headings, empty states, error messages, placeholder text, tooltips.
