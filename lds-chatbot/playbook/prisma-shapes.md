# Prisma Query Shapes Rule

## Rule

Never hand-write interfaces for Prisma model data. Define the exact shape a query returns using the `satisfies` operator (Prisma v7) or `Prisma.validator` (Prisma v5/v6), derive the TypeScript type from that shape with `Prisma.XxxGetPayload`, and spread the shape directly into every query that uses it. One definition, one type, always in sync.

> **Prisma v7 note**: `Prisma.validator` was removed in v7. Use `satisfies Prisma.XxxDefaultArgs` instead — type inference with `Prisma.XxxGetPayload` works identically.

---

## Forbidden — Hand-Written Interfaces

```ts
// ❌ Forbidden — manually maintained, drifts silently when schema changes
export interface Agent {
  id: string
  userId: string
  name: string
  isPublic: boolean
  createdAt: Date | null
  updatedAt: Date | null
  workflowId: string
  systemPrompt: string
  steps: unknown
  model: string | null
  // ... every new column added to the schema must be added here by hand
}

// ❌ Forbidden — $inferSelect is the Drizzle pattern; Prisma has a purpose-built mechanism
export type AgentSelect = typeof agents.$inferSelect
```

Interfaces like these are wrong in two ways. First, they must be kept in sync with `prisma/schema.prisma` by hand — no tool enforces this, so drift is guaranteed. Second, the query that fetches the data has no compile-time connection to the interface, so adding a `select` clause that omits a field silently breaks the type.

## Correct — Prisma Shape Pattern

```ts
import { Prisma } from "@/lib/generated/prisma/client"

// Prisma v7 — use `satisfies` (Prisma.validator removed in v7)
export const agentWithSubagents = {
  include: {
    subagents: true,
  },
} satisfies Prisma.AgentDefaultArgs

// Prisma v5/v6 — use Prisma.validator (still works if on older version)
// export const agentWithSubagents = Prisma.validator<Prisma.AgentDefaultArgs>()({
//   include: { subagents: true },
// })

// Derive the type — automatically updates when the shape or schema changes.
// If you add a relation to the include, AgentWithSubagents gains it automatically.
export type AgentWithSubagents = Prisma.AgentGetPayload<typeof agentWithSubagents>

// Minimal shape for list views — only what the UI needs.
// Adding a new column to the schema does NOT widen this type; you opt in explicitly.
export const agentSummary = Prisma.validator<Prisma.AgentDefaultArgs>()({
  select: {
    id: true,
    name: true,
    isPublic: true,
    userId: true,
    model: true,
    createdAt: true,
  },
})
export type AgentSummary = Prisma.AgentGetPayload<typeof agentSummary>
```

Rules:
- Import `Prisma` (the namespace) from `@/lib/generated/prisma/client`, not from `@prisma/client`
- The validator generic is always `Prisma.XxxDefaultArgs` where `Xxx` is the model name
- The derived type is always `Prisma.XxxGetPayload<typeof shapeName>`
- Shape and derived type are always exported together, adjacent in the file
- Shape names are camelCase, descriptive of what they include (`agentWithSubagents`, `agentSummary`)
- Derived type names are PascalCase, matching the shape name (`AgentWithSubagents`, `AgentSummary`)

## Using Shapes in Queries

The shape object spreads directly into the Prisma call. The query never re-declares `select` or `include` inline.

```ts
// models/agents/queries.ts
import "server-only"

import { prisma } from "@/lib/db"
import { agentWithSubagents, agentSummary } from "./schema"
import type { AgentWithSubagents, AgentSummary } from "./schema"

export async function getAgent(id: string, userId: string): Promise<AgentWithSubagents | null> {
  return prisma.agent.findFirst({
    where: { id, userId },
    ...agentWithSubagents,   // spreads include: { subagents: true }
  })
}

export async function getAgents(userId: string): Promise<AgentSummary[]> {
  return prisma.agent.findMany({
    where: { userId },
    ...agentSummary,         // spreads select: { id, name, isPublic, ... }
  })
}
```

Rules:
- Spread the shape constant into the query — never copy its fields inline
- Return types are annotated with the derived `GetPayload` type, not `any` or a hand-written interface
- If a query needs a shape used nowhere else, define it in `queries.ts` with a `const` + `type` pair, not inline

## What Lives in `models/*/schema.ts` After This Rule

```ts
// models/agents/schema.ts

import { Prisma } from "@/lib/generated/prisma/client"

// Domain constants — NOT generated by Prisma, defined here explicitly
export const AGENT_STATUS = {
  ACTIVE: "active",
  ARCHIVED: "archived",
} as const
export type AgentStatus = (typeof AGENT_STATUS)[keyof typeof AGENT_STATUS]

// Named query shapes + derived types
export const agentWithSubagents = Prisma.validator<Prisma.AgentDefaultArgs>()({
  include: { subagents: true },
})
export type AgentWithSubagents = Prisma.AgentGetPayload<typeof agentWithSubagents>

export const agentSummary = Prisma.validator<Prisma.AgentDefaultArgs>()({
  select: {
    id: true,
    name: true,
    isPublic: true,
    userId: true,
    model: true,
    createdAt: true,
  },
})
export type AgentSummary = Prisma.AgentGetPayload<typeof agentSummary>
```

Rules:
- Domain const enums (`DATASET_STATUS`, `ENTRY_STATUS`, `MCP_TRANSPORT`) belong here — they are not generated by Prisma and must be defined explicitly
- No hand-written interfaces for model data
- No duplicated type aliases
- `schema.ts` does not import from `queries.ts`, `service.ts`, or anything in `app/`

## `select` vs `include`

**Use `select` when:**
- Feeding a list view, API response, or hook that only needs a controlled subset of fields
- You want a stable contract: new columns added to the schema are NOT included until you explicitly opt in
- Minimising data over the wire matters (large tables, frequent queries)

**Use `include` when:**
- A service operation needs relations alongside all scalar fields
- A detail view requires the full record plus related rows
- `include` with no `select` = all scalar fields + the specified relations (Prisma default behaviour)

Never mix `select` and `include` at the top level of the same shape — Prisma's type system handles nested selects inside an `include`, but a top-level `{ select: ..., include: ... }` is a type error.

---

## Schema Changes Are Automatic

When you add a column to `prisma/schema.prisma` and run `npx prisma migrate dev`:

1. The Prisma client regenerates in `lib/generated/prisma/`
2. Shapes using `include` (no top-level `select`) automatically include the new column — their `GetPayload` type widens without any code change
3. Shapes using `select` keep their fixed subset — the new column is absent until you explicitly add `newColumn: true` to the shape
4. TypeScript catches any inconsistency at compile time — a shape that references a deleted column becomes a type error immediately

This is the intended behaviour. `include` shapes are permissive by design. `select` shapes are explicit contracts — adding a column to the DB does not silently expose it through an API response.

## Forbidden Patterns

```ts
// ❌ Hand-written interface that duplicates schema fields
export interface AgentRow {
  id: string
  userId: string
  name: string
}

// ❌ Inline select inside the query — shape is not reusable, type is not derived
const agents = await prisma.agent.findMany({
  where: { userId },
  select: { id: true, name: true, model: true },
})
// → the return type is inferred locally and cannot be imported by routes or hooks

// ❌ Casting the return type instead of deriving it
const agent = await prisma.agent.findFirst({ where: { id } }) as Agent
// → the cast hides shape mismatches at compile time

// ❌ Defining the shape without Prisma.validator
export const agentShape = {
  include: { subagents: true },
}
// → literal types are widened; Prisma.AgentGetPayload<typeof agentShape> produces `unknown`

// ❌ Importing Prisma from the npm package instead of the generated client
import { Prisma } from "@prisma/client"
// → use "@/lib/generated/prisma/client" so the project's generated types are used
```

---

## Relation to Other Rules

- This rule extends `models.md`: the `schema.ts` file's responsibility is now "define the named shapes this model's queries use". Hand-written interfaces are removed; `Prisma.validator` shapes replace them.
- `queries.ts` spreads shapes directly into Prisma calls and annotates return types with `GetPayload` types imported from `./schema`. No inline `select` or `include` in query functions.
- Response types exported from `models/*/schema.ts` travel to route handlers and from there to hooks with no intermediate redefinition. See `api-routes.md`.
- `lib/db/schema.ts` is a barrel that re-exports domain constants (`DATASET_STATUS`, `ENTRY_STATUS`, etc.) from each model's `schema.ts`. It does not define shapes — shapes live in their model directory.
